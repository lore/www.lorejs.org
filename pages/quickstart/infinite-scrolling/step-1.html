--- layout: quickstart permalink: /quickstart/infinite-scrolling/step-1/ --- {% capture markdown %} # Step 1: Add Infinite Scrolling In this step we'll add infinite scrolling to our Feed. > You can view the finished code for this step by checking out the `infinite-scrolling.1` branch of the [completed project](/quickstart/misc/completed-project/). ### Approach Infinite Scrolling is similar to pagination except for two key points: 1. Instead of letting the user select a page, they can only load the next page 2. Instead of displaying one page at a time, the results are all appended to single running array It's important to point out that these are _view specific concerns_; meaning nothing about them requires changes to infrastructure or to the API. They're all about how a user interacts with data, and have nothing to do with how data is fetched or stored. To that end, this section will be focusing on how to implement infinite scrolling in a simple and reusable way. ### Add NextPage to Meta Property Unlike traditional pagination links, where we need to calculate the number of links early on, infinite scrolling only cares about whether there is a "next" page. So we're going to provide that information by adding another property to the `meta` field of the `tweets` collection. Open up `config/connections.js` and update the collection's `parse` method to add the `nextPage` property from the API response to to the `meta` data. This field will either contain the number of the next page of data or be null if there are no more pages to display. ```js ... parse(attributes) { this.meta = { totalCount: attributes.meta.paginate.totalCount, perPage: attributes.meta.paginate.perPage, nextPage: attributes.meta.paginate.nextPage }; return attributes.data; } ... ``` ### Add InfiniteScrolling Decorator InfiniteScrolling has _a lot_ of boilerplate associated with it. Components that implement it need to: * fetch the first page of data * fetch the next page if the user requests it * provide some kind of data that signals a new page is being fetched * combine all of the data into a single array containing all pages of data The logic for that can also be a little tricky, but luckily it's easy enough to encapsulate into a decorator that can be applied to any component.
