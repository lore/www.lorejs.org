--- layout: features permalink: /features/config/actions/ --- {% capture markdown %} # config/actions.js This file is connected to the `lore-hook-actions` hook and overrides the default action behaviors. ## Purpose This config is currently only relevant if your app requires the use of WebSockets while performing optimstic updates, as this use case requires sending a unique client-side generated ID to the server. In addition to the configuration options below, there are also plans to support disabling the blueprints and allowing you to provide a custom override for each blueprint without having to override the entire `lore-hook-actions` hook. ## Example Config File ```js module.exports = { addCidToBody: false, cidBodyAttributeName: 'cid' }; ``` ## Configuration Options #### addCidToBody Determines whether the client-side id (the cid attribute in a model) is sent to the server when creating data. This is not necessary when interacting with REST APIs as the actions can associate the associate the request and response, but when performing optimistic updates using WebSockets it is neccesary to send a client-generated ID to the server in order to associate the data created on the client side with the data broadcasted by the server. #### cidBodyAttributeName This field let's you change the name of the cid field sent to the server when performing optimistic updates through WebSockets. By default the property send to the server is called `cid`, matching the name of the attribute on the client-side, but you can change if you need to to match the API. {% endcapture %} {{ markdown | markdownify }}
